1차 MVP 기준 포함 스택은 다음으로 확정하는 게 맞다.

```text
OmniOne CX
→ 모바일 외국인등록증 / 모바일 신분증 인증 진입점

OpenDID
→ Housing Pass VC 발급·VP 제출·검증

OmniOne Chain
→ consentHash / verificationHash / reviewHash 기록

AI 계약 리뷰 모듈
→ 월세계약서 분석, 다국어 요약, 체류기간-계약기간 정합성 경고

Web App Stack
→ Next.js / React / TypeScript / Backend API / DB

Smart Contract Stack
→ Solidity / ethers.js
```

Sui, Walrus, PTB, zkLogin은 1차 MVP에 포함하지 않고 2차 고도화 스택으로 분리하는 게 맞다. Sui 바운티 문서도 Sui 기술 스택, zkLogin, PTB, Walrus 등은 별도 요건으로 제시하고 있으므로 1차 범위에 억지로 넣지 않는 편이 안정적이다.

## 2.2 문서에서 확인된 핵심 내용

OmniOne CX API 매뉴얼은 OmniOne CX가 VC-Verifier와 통합 인증 솔루션으로 구성되어 있고, 모바일 신분증을 제출하는 API를 설명한다고 한다. 또한 연동 흐름은 **토큰 생성 → 인증 요청 → 상태 확인 → 결과 요청 → 토큰 파싱** 순서다. 

1차 MVP에서 봐야 할 API는 다음이다.

| 기능                      | API                                      |
| ----------------------- | ---------------------------------------- |
| 인증사 목록 조회               | `GET /oacx/api/v1.0/provider/list`       |
| 최초 Token 생성             | `POST /oacx/api/v1.0/trans`              |
| QR 인증 요청                | `POST /oacx/api/v1.0/authen/qr/request`  |
| AppToApp/WebToApp 인증 요청 | `POST /oacx/api/v1.0/authen/app/request` |
| QR 검증 요청                | `POST /oacx/api/v1.0/authen/qr/result`   |
| AppToApp/WebToApp 검증 요청 | `POST /oacx/api/v1.0/authen/app/result`  |
| 결과 Token 파싱             | `POST /oacx/api/v1.0/trans/token`        |

외국인증 제공 데이터는 매뉴얼 부록에 별도 표로 있고, 핵심 key는 다음이다. 

| key             | 의미             | SettlePass claim 변환                      |
| --------------- | -------------- | ---------------------------------------- |
| `title`         | 외국인등록증         |                                          |
| `engtitle`      | RESIDENCE CARD |                                          |
| `qualifstaycd`  | 체류자격코드         | 1차에서는 원문 저장 금지                           |
| `qualifstaynm`  | 체류자격명          | 1차에서는 원문 저장 금지                           |
| `staydate`      | 체류허가일자         | 필요 시 내부 계산용                              |
| `stayexdate`    | 체류만료일자         | `residenceValid`, `residenceExpiryMonth` |
| `stayregloc`    | 체류확인 지역        | `regionLevel1`                           |
| `stayregdate`   | 주소지 신고일        | 1차에서는 비공개                                |
| `provider`      | 기관 id          | `coresidence`                            |
| `resultCode`    | 결과 코드          | 인증 성공 여부                                 |
| `clientMessage` | 결과 메시지         | 인증 결과 표시                                 |
| `signType`      | 인증종류           | `ENT_MID`                                |

## 2.3 1차 MVP 적용 방식

1차에서는 실제 외국인등록증 실연동을 보장하지 않는다. 구조는 반드시 **OmniOne CX Adapter**로 만들고, 실행 모드는 두 개로 나눈다.

| 모드             | 설명                                              |
| -------------- | ----------------------------------------------- |
| `CX_REAL_MODE` | OmniOne CX 실제 API 연동. 테스트 credential 제공 시 사용    |
| `CX_MOCK_MODE` | OmniOne CX 응답 구조를 모사한 mock claim 생성. 예선 시연 안전장치 |

1차 MVP claim 변환:

```json
{
  "identityVerified": true,
  "credentialType": "MOBILE_FOREIGNER_ID",
  "ageOver19": true,
  "residenceValid": true,
  "residenceExpiryMonth": "2026-12",
  "regionLevel1": "Seoul"
}
```

## 4.4 1차 MVP 적용 방식

1차에서 OpenDID는 전체 운영환경을 완전 구축하는 것이 아니라 다음 범위로 제한한다.

```text
CX Adapter claim
→ HousingPassCredential VC 발급
→ 임대인 요청
→ 사용자 동의
→ HousingPass VP 제출
```

# 7. AI 계약 리뷰 모듈 공식 문서

1차 MVP의 AI는 “부동산 공증”이 아니라 **외국인이 계약서 내용을 이해하고 최종 확인하는 보조 기능**이다. 따라서 블록체인에 남기는 값도 `documentHash`가 아니라 `reviewHash`다.

```text
reviewHash = hash({
```

# 10. 공식 출처 기준 구현 우선순위

## 1순위: OmniOne CX Adapter

근거: 1차 MVP의 첫 단계는 모바일 외국인등록증 인증이다. 최종 PPT에서도 “모바일 외국인등록증 인증 → OmniOne CX 본인확인 → 성인·체류 유효 claim 추출”로 설계되어 있다. 

읽을 문서:

1. OmniOne CX API 매뉴얼
2. 모바일 신분증 개발지원센터
3. 테스트베드 App 설치 가이드

구현 범위:

```text
provider list
token 생성
QR/WebToApp request
result
token parsing
CX_MOCK_MODE fallback
```

## 2순위: OpenDID Housing Pass

근거: OpenDID는 1차 MVP에서 선택적 정보공개와 Housing Pass VC/VP의 핵심 레이어다.

읽을 문서:

1. did-doc-architecture
2. did-demo-server
3. did-demo-app
4. W3C VC Data Model

구현 범위:

```text
HousingPassCredential JSON
HousingPassPresentation JSON
임대인 Verifier Portal
검증 결과 Yes/No
```

## 3순위: OmniOne Chain Audit Log

근거: 1차 MVP에서 블록체인에 남길 것은 계약서 원문이 아니라 동의·검증·리뷰 확인 이력이다.

읽을 문서:

1. 2026 해커톤 가이드북 OmniOne Chain 장

3. ethers.js docs

구현 범위:

```text
consentHash
verificationHash
reviewHash
txHash 표시
원문 개인정보 저장 금지
```

## 4순위: AI 계약 리뷰

근거: 외국인 정주 불편의 핵심은 계약서 이해와 절차 이해다. AI는 계약서의 핵심 조건, 위험 조항, 체류기간-계약기간 정합성을 요약한다.

읽을 문서:

1. OpenAI API Docs
2. Responses API
3. Structured Outputs
4. File Inputs

구현 범위:

```text
PDF/text input
계약 핵심 조건 추출
위험조항 탐지
다국어 요약
JSON Schema 고정
사용자 최종확인
reviewHash 생성
```

| OmniOne Chain 전용 최신 공개 개발자 포털  | 공개 검색 기준 명확한 별도 문서 확인 안 됨                                   | 운영진/멘토에게 콘솔 URL, Chain ID, RPC, API Key 문서 요청 |
